## src/ vs build/

`src/` holds the human-authored source. `build/` holds the compiled output produced by `npm run build` (via `@wordpress/scripts` / webpack).

WordPress always loads from `build/` вЂ” the plugin's PHP registers the block from `MONOLITH_PATH . 'build'`.

Never edit files in `build/` directly. They are overwritten on every build run.

---

## Key file responsibilities

### `src/view.js`
The entire front-end brain. Registers the Interactivity API store (`monolith` namespace) which:
- Listens for `wc-blocks_viewed_product` to open the modal
- Fetches product data from the WooCommerce Store API
- Manages the View Transitions animation
- Handles URL hash routing
- Manages focus trap and keyboard events
- Locks / unlocks body scroll
- Syncs `aria-hidden` on the page wrapper

### `src/render.php`
Outputs the modal HTML shell on every page that contains the block. Sets the Interactivity API server-side state and config. All actual content is injected by `view.js` after the Store API fetch.

### `src/block.json`
Declares the block and its `blockHooks` entry вЂ” this is what causes the modal to auto-insert as the last child of every `woocommerce/product-collection` block without any editor action.

### `includes/class-monolith-plugin.php`
Runs on `init` to register the block (from `build/`), the front-page template, and the three block style variations. Also hooks into `wp_speculation_rules_href_exclude_paths` to prevent WordPress 6.8's prerender from firing on product pages before a click.
